01 整体概览

知更是一个视频创作者的 AI 工作台。你给它一段脚本，它帮你分析、生成 B 站/抖音/小红书/YouTube 的标题、标签、描述，还能爬取竞品数据做分析。

一句话理解架构

用户说话 - AI 思考 - 调用工具(爬数据/分析/生成) - 把结果存起来 - 回复用户

02 文件结构

把项目想象成一栋楼：前端是门面，Agent是大脑，工具是手脚，数据库是记忆。

knower-agent/ < Agent 的家 agent/ < 大脑 core.js < 核心循环 (while + LLM) state.js < 状态机 (知道自己在哪步) router.js < 路由器 (决定下一步) processor.js < 数据处理器 (工具结果 - 结构化) cache.js < 缓存 (5分钟TTL) tools/ < 工具箱 crawl_data.js < 爬一个平台 crawl_data_batch.js < 同时爬多个平台 (并行) query_data.js < 查数据库 analyze_script.js < 分析脚本 expand_script.js < 生成各平台物料 save_result.js < 存到数据库 + 提炼记忆 suggest_topics.js < 选题建议 (内部调LLM) request_user_input.js < 弹表单问用户 llm/ < 翻译官 (和AI模型对话) index.js < 根据provider选客户端 anthropic.js < Claude API 封装 openai-compat.js < OpenAI 兼容API封装 db/ < 记忆库 (SQLite) crawler/ < 爬虫 (Python, MediaCrawler) config/index.js < 读配置 run.js < 命令行测试入口

03 核心循环

Agent 的核心就是一个 while 循环，每次循环做三件事：问 AI、执行工具、把结果告诉 AI。直到 AI 说"我说完了"才停。

主循环流程图

用人话说一遍

想象你在和一个很聪明但没有手的助手对话:

1. 你给助手一个任务("帮我分析这个UP主")

2. 助手说:"我需要先查一下他的数据"(决定调工具)

3. 你替助手去查了数据(执行crawl_data)

4. 你把查到的数据告诉助手(塞回messages)

5. 助手看到数据后说:"分析完了,给你结论"(输出文本)

6. 或者助手说:"还得再查一个东西"(再调工具) - 回到第3步

关键点: 助手(LLM)决定做什么,你(代码)负责执行。

04 状态机

之前Agent是"无脑循环",现在加了状态机,它知道自己处于什么阶段,下一步该干什么。

状态转换图

防止跳步

没有状态机时,AI可能跳过分析直接生成物料。有了状态机,router.js会检查:"你现在在analyzing阶段,不能调expand_script"

知道做完了什么

状态机记录了crawlData、analysis、materials等数据。AI不用每次都问"我爬过数据了吗",上下文会告诉它

可恢复

状态可以序列化保存。如果中途崩溃,重启后能从断点恢复(架构已支持,待实现)

LangChain是什么?

LangChain是一个AI应用开发框架,把"调用LLM"抽象成积木系统:

  • Chain(链): 把多个步骤串起来,像流水线
  • Agent(代理): 让LLM自己决定调什么工具
  • Tool(工具): 可复用的功能模块
  • Memory(记忆): 管理对话历史
  • Retriever(检索器): 从文档中找相关信息

优点: 生态丰富,几千个集成,社区活跃

缺点: 抽象层太多,debug困难,版本升级经常breaking change

LangGraph是什么?

LangGraph是LangChain团队的下一代框架,用图(Graph)描述工作流:

  • Node(节点): 每个处理步骤是一个节点
  • Edge(边): 节点之间的连接,可以是条件边
  • State(状态): 所有节点共享一个TypedDict状态
  • Checkpoint: 每一步都自动保存,可以断点续跑
  • Human-in-the-loop: 可以在任意节点暂停等人类确认

优点: 状态管理强,支持复杂工作流,可观测性好

缺点: 依赖重,学习曲线陡,对桌面应用来说过于复杂

09 总结

结论: 知更Agent不差,只是不同

知更Agent是一把精巧的瑞士军刀,LangChain/LangGraph是一套工业级的流水线设备。

你不需要用流水线设备来削苹果。

知更做对了什么

  • 代码量小(核心~500行),每个文件职责清晰
  • 零框架依赖,不会被版本升级坑
  • 状态机+路由器,比纯while循环聪明
  • 并行爬取,效率不输LangGraph
  • 记忆系统,越用越懂用户
  • 桌面端部署,用户数据不出本地

还可以更好

  • Human-in-the-loop还是弹表单,不是真正的中断恢复
  • 可观测性只有console.log,没有可视化追踪
  • 状态序列化有了,但断点续跑还没实现
  • 工具输入校验不够严格
  • 没有测试,重构容易引入回归

什么情况该上LangChain/LangGraph

  • 需要对接20+个外部服务
  • 需要RAG(检索增强生成)
  • 需要多人协作的复杂审批工作流
  • 需要生产级的可观测性和错误恢复
  • 团队已经熟悉Python生态
